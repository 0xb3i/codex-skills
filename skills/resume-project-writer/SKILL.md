# 简历项目与面试写作

将简历正文、完整项目叙事和面试 QA 视为三层不同交付物：正文负责制造高质量 hook，项目文档负责保证机制自洽，QA 负责承受连续追问。

## 必读路由

- 写或改简历短文本、项目 bullet、量化结果：读取 [references/writing-guidelines.md](references/writing-guidelines.md)。
- 扩写完整项目、构造或吸收面试 QA：同时读取 [references/project-and-qa-workflow.md](references/project-and-qa-workflow.md)。
- 用户要求前沿依据、论文素材、严谨引用，或问题涉及 LLM/Agent/RL 新方法：读取 [references/knowledge-sources.md](references/knowledge-sources.md)，按需检索两个飞书知识库。
- 修改飞书简历文档：同时遵循已安装 `lark-doc` / `lark-wiki` skill；写前局部 fetch，写后回读验证。

## 核心工作流

1. **恢复事实边界**：确认真实系统的组件、训练方式、数据来源、指标和用户实际承担的工作。不要为了术语完整而虚构 Planner、Generator、Reranker 等不存在的节点。
2. **识别真正难点**：优先寻找任务特有的 failure mode、数据瓶颈、验证难题和工程约束，而非复述业界共识。
3. **分层写作**：
   - 简历正文：一句总起 + 3–5 个高密度 bullet + 量化结果。
   - 完整项目：背景、数据、训练、reward/优化、评测与上线闭环。
   - 面试 QA：围绕每个 hook 补充为什么、如何实现、如何验证、有什么局限。
4. **因果表达**：优先使用“针对具体问题，采用具体机制，获得或验证具体结果”。算法名本身不需要通用动机；任务特有设计必须说明解决了什么问题。
5. **吸收反馈**：用户以面试官身份追问时，先直接回答，再把问题和精炼答案纳入对应章节的 QA；同步检查它是否暴露了简历正文或项目背景的错误重心。
6. **验证一致性**：检查架构口径、数据数量、指标增益和正文机制能否互相对应；飞书写回后必须局部回读。

## 输出原则

- 简历是钩子，不是论文。删除面试官已知的通用知识，保留项目特有设计。
- 不罗列技术名词。每个机制都要对应真实 failure mode 或约束。
- 不用未经用户确认的数字冒充事实；占位数字必须显式标记。
- 不因用户要求“包装”就脱离其熟悉领域。可以升级叙事，但必须能承受三层追问。
- 用户给出模板时，沿用其信息密度与 QA 深度，不机械复制其具体结构或过时结论。
- 面试问答允许比简历正文更详细，但仍优先给出清晰、可复述的主线。

## 知识库使用

将知识库视为素材与证据来源，不视为自动正确答案。检索后：

1. 选择与当前项目机制或追问直接相关的页面。
2. 区分论文原始结论、知识库作者解读和为当前项目做的合理推演。
3. 不把论文方法硬塞进简历；只有能解释真实设计、对比选型或补强 QA 时才引用。
4. 引用具体论文或结论时，保留标题、关键结论及适用边界；必要时再核对原论文。

## 交付

- 用户只要短文本：输出可直接复制的完整项目条目，并简要指出主要 hooks。
- 用户要完整材料：提供项目叙事和分章节 QA；若目标是飞书文档，直接写回并给出链接。
- 用户进行模拟面试：以面试回答为主，同时把新问题沉淀到文档或建议的 QA 条目中。
- 每轮修改说明保持简短，重点交付更新后的成品。
