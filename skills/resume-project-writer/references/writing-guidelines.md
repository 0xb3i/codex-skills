# 技术简历写作准则

## 1. 简历是 hook

- 每个短语用于引导面试官追问，不负责把完整原理讲完。
- 正文写“做了什么、项目特有地怎么做、结果如何”。通用动机、完整推导和踩坑留给 QA。
- 每个 hook 至少准备三层追问：为什么、怎么实现、如何证明有效或有什么局限。

## 2. 推荐结构

1. 项目名：短名词短语。
2. 总起句：业务场景 + 技术方案骨架 + 交付结果。
3. 3–5 个 bullet：数据构建、训练/算法、任务特有机制、工程优化、效果评估。
4. 效果：只保留能与正文机制对应的核心指标。

## 3. 信息密度规则

### 删除业界共识

不要解释“RL 相对 SFT 的一般价值”“RAG 为什么需要检索”“Agent 为什么需要工具调用”等面试官已知内容。直接写具体实施。

### 机制必须对应 failure mode

不要写“设计复合 Reward”。应写清任务中的偷懒或作弊路径，以及对应约束。例如：

- 万金油式罗列根因 → 集合级 F1 摊薄 precision。
- 只停留在上层指标 → 按根因定位深度分级给分。
- 依赖先验编造结论 → 证据溯源，结论必须对应真实 observation。
- 空转或重复调用 → 步数成本与重复调用惩罚。

### 使用“问题 → 措施”而非术语罗列

推荐：`针对模型易出现 A、B 的问题，设计 X、Y；针对 C 约束，引入 Z。`

算法本身若无需解释通用动机，可直接写：`采用 GSPO 进行强化学习；针对……设计……。`

## 4. 尊重真实架构

- 若系统是端到端 Agent，不要擅自拆成 Planner / Generator。
- 只有真实存在独立模型、节点、训练目标或接口边界时，才使用模块名称。
- “planning”可以描述行为过程，不等同于系统存在名为 Planner 的组件。
- 修改一个架构口径后，全局搜索标题、摘要、架构、QA、核心贡献和未来优化，统一替换。

## 5. 数据叙事原则

- 判断真正的泛化单位。对于诊断 Agent，同一 Query 在不同环境状态下可能对应不同根因与轨迹，环境数据往往比同义 Query 改写更重要。
- 不把文本扩写数量包装成核心数据亮点，除非语言覆盖确实是主要瓶颈。
- RLVR 数据重点说明：环境、Groundtruth、可验证证据、难度分层、多路径可解性和自动校验。
- 固定模板只应约束业务合法性，不应规定唯一数值和唯一 trajectory。

## 6. 语言与格式

- Bullet 使用：`环节标签：分句一；分句二；分句三。`
- 不写元话语、读者旁白或过多括号。
- 不生造术语；优先使用领域标准行话。
- 中英混排可保留 Query、Groundtruth、Teacher Model、CoT、RLVR、reward、observation、loss mask 等常用术语。
- 一句话过长时，优先删除通用信息，而不是堆叠更多连接词。

## 7. 量化指标

- 使用“从 base 的 X 提升至 Y（+Zpt）”。
- 只比较 base 和最终版本，除非中间阶段本身是重要实验结论。
- 指标必须可追溯到正文机制：步数下降对应调用成本，证据准确率对应溯源约束。
- 未确认数字不得写成事实；建议区间必须标记为占位。

## 8. 自检清单

- 是否出现不存在的系统模块？
- 是否把业界共识写进了有限篇幅？
- 是否每个技术名词都对应具体问题？
- 数据亮点是否抓住了真正的泛化维度？
- 是否存在数字互相矛盾或无法解释？
- 面试官连续追问三层时，用户能否回答？
