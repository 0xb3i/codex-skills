# Weekly Report

## 目标

把零散素材整理成面向 leader 的中文周报段落。输出要让读者快速理解：为什么做、做了什么、结果如何、下周继续做什么。

默认不要写成流水账，也不要照抄素材。优先产出可以直接粘贴到周报里的成段文本。

## 输入处理

1. 先识别素材中的项目名、背景/目的、动作、结果指标、风险、依赖、下周计划。
2. 多个事项按项目或主题分组；同一项目下把重复信息合并。
3. 指标优先保留原始口径，不擅自改写含义；缺少口径时用保守表述。
4. 遇到明显不确定、口径未确认或数据未最终对齐的信息，用“当前观察”“初步数据”“待对齐”标明，不包装成确定结论。
5. 如果素材只有动作没有结果，仍可输出进展，但要显式说明当前结果状态，例如“数据仍在累积中”“效果待实验验证”。

## 输出结构

默认输出两个部分：

```markdown
本周进展

<按项目写 1-3 段，每段围绕目的、动作、结果展开。>

next_step

- <下周动作 1>
- <下周动作 2>
```

当用户只给一个事项时，“本周进展”下输出一段即可；当用户给多个事项时，每个项目使用短标题加一段正文。`next_step` 单独放在最后。

## 段落写法

每个项目优先使用这个骨架，但不要机械套模板：

> 为了 <让 leader 明白目的/背景>，本周我们 <简短说明关键动作和范围>。目前 <用指标或状态说明结果>，其中 <绝对指标 + 相对变化/对比基线/实验结论>；如果结果仍未定稿，补充 <待观察/风险/依赖>。

写作要求：

- 第一层信息先讲业务目的，不先堆实现细节。
- 动作用一句话讲清，保留关键技术名词、实验名、版本、时间窗口。
- 结果尽量“指标说话”：绝对值 + 相对变化优先，例如 `score 0.642 -> 0.849（+0.229/+36.7%）`。
- 有显著性、置信区间、p 值时保留；不显著时明确写“不显著”或“持续观察”。
- 有风险时放在段末，不单独渲染成很重的风险段，除非用户明确要求。
- 避免夸大收益；不要把“开发完成”“联调中”写成“已取得业务收益”。
- 保留必要英文专有名词和指标名，例如 `planner`、`verifier`、`TTFT`、`uauc`、`GMV`。

## next_step 写法

`next_step` 只放未来动作，避免重复本周进展。每条以动词开头，写清目标和验收信号。

优先顺序：

1. 继续观察/复盘实验数据。
2. 完成联调、上线、推全、反转实验或 LR。
3. 补齐数据、修复样本/口径问题、扩大评测集。
4. 处理明确风险或跨团队依赖。

## 缺信息处理

不要频繁追问。能合理整理时直接输出；同时用括号标出缺失信息：

- `（指标待补充）`
- `（实验时间窗口待确认）`
- `（owner 待补充）`
- `（结论待 PM/DS 对齐）`

只有在无法判断项目边界、素材互相矛盾、或结果会被误导性表达时，才向用户提出一个简短问题。

## 示例

输入：

```text
榜单预检，候选商品不足 20 个会导致榜单工具空结果。新增启动阶段类目成榜预检，只注入可成榜类目。开发中。下周联调。
planner SFT，5k 数据，score 0.642 到 0.849，tool_exact_rate 0.348 到 0.745。
```

输出：

```markdown
本周进展

为了降低选品 Agent 在榜单类目候选不足时调用空结果工具的概率，本周新增了启动阶段的类目成榜预检机制：系统会提前检查二级类目的成榜状态，仅将可成榜类目注入 Agent 上下文。目前逻辑仍在开发中，后续需要通过联调验证对空结果率和可用性的改善效果。

为了提升 planner 的工具规划能力，本周基于首批 5k+ 压测数据完成 Gemma26B 的全量 SFT。首轮结果显示整体 score 从 0.642 提升到 0.849（+0.229/+36.7%），tool_exact_rate 从 0.348 提升到 0.745（+0.397/+114%），主要收益来自工具选择更贴近 gold truth，并减少了不必要的额外工具调用。

next_step

- 完成榜单预检逻辑联调，验证对空结果工具调用的改善效果。
- 重跑训练数据，修正榜单轨迹和 carveout 样本损失问题，并扩大后续训练/评测数据集。
```
