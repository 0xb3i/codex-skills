# Polishing Feishu Docs

## Overview

把已有飞书原稿或零散素材烹饪成一份能读、好读、适合汇报/协作的文档：结论先行、事实不丢、结构清楚、表达克制、版式服务阅读。先读原文、用户要求和评论，再用 `lark-doc` / `lark-drive` / `lark-whiteboard` 做最小必要编辑。

## Workflow

1. 读取素材：使用 `lark-doc` 读取正文；编辑前按它要求读取 shared、fetch、XML、style、update workflow 引用。若文档内有 sheet/base/whiteboard token，按 `lark-doc` 路由下钻，不只展示标签。
2. 读取评论：若用户提到 remark / 评论 / 批注 / review，或文档明显需要吸收外部反馈，按 `lark-drive` 评论规则读取未解决评论；需要定位正文时使用 docx 评论位置能力。把带评论 relation 的 block 标为评论锚点区。
3. 做融合审计：先判断现有正文是否只是 raw 素材、评论和原文是否重复/冲突、吸收评论后是否需要调整章节顺序或删改旧段落；不要逐条把评论贴进原结构。若新增指标、维度、定义或口径，必须全局扫描并同步所有依赖位置（结论、公式、结果表说明、风险、下一步、案例解读），不能只改评论附近。
4. 搭结构：把用户给的素材、位置提示和评论要求归并成 3-5 个顶层章节；素材只是“放了一些东西”时，不保留草稿顺序，按读者理解路径重排。
5. 提炼结论：找出最应该让老板先看到的 1-3 条结论。只有确有结论时，在文档最前面放一个橙色大头钉提示块，用“结论：”开头，一句主判断后接 1-2 条关键依据或风险。
6. 重写正文：每段按“为了...，我们/建议/结论是...；原因/例子是...”组织。先讲目的和动作/结论，再展开依据；不要编造原文没有的事实。不确定信息用“待确认：...”或占位符保留。
7. 吸收评论：把能确定的评论融入正文、表格、结论或下一步，并同步删改被评论取代的旧表达；有争议或信息不足的评论标成待确认，不把评论原文堆成附录，也不把已回复内容重复制造噪音。
8. 规范格式：统一标题层级、列表、表格、行内 code、高亮、callout、引用块和画板。优先改现有块；只有结构明显混乱时再整段替换。
9. 写回文档：用 `lark-cli docs +update` 的最小可控命令。带评论锚点的 block 可以局部改写，但必须保留评论对应的完整引用内容或可识别承载文本；不要删除该 block、替换为空、整块重建成无关内容，或移动到会破坏评论上下文的位置。大幅重构时优先在相邻 block 插入融合内容，保留原评论锚点 block 承载评论。替换或删除 block 后重新 fetch，再继续依赖新 block id。

## Style Contract

| 场景 | 处理 |
|---|---|
| 核心结论 | 文档顶部橙色大头钉提示块；“结论：”+ 一句主判断 + 1-2 条 bullet；核心增量可用黄色背景高亮 |
| 表格 | 使用 `<thead>/<th>`；表头使用浅蓝填充背景 `rgb(225,234,255)`；不要额外给表头包 `<b>`；列名简短，数值右对齐 |
| 技术字段 | 仅 `board_param_score` 这类 snake_case 字段、接口名、参数名、枚举值、路径、命令、工具名使用行内 code；普通术语、缩写和数字不要用 code |
| 重要但非最终结论 | 用高亮或引用块，不要堆多个提示块 |
| 评论吸收 | 可确定的评论改进直接进入正文/表格/下一步；未确认信息写“待确认：...”；不要单独堆“评论整理”章节，除非用户要求 |
| 评论锚点保护 | 带评论 relation 的 block 可小幅编辑，但必须保留评论对应的完整引用内容或可识别承载文本；不要整块删除、清空或改成无关内容 |
| Raw 素材 | 先抽象成读者需要的判断、表格或案例解读；只保留支撑结论所需的原始代码/日志，不把整段素材原样留下 |
| 章节层级 | 顶层章节控制在 3-5 个；错误分布、指标拆解等从属内容放到对应结果章节下，不要都升成 h1 |
| 流程/复杂逻辑 | 简单流程用编号列表；跨角色、分支多、依赖多时使用画板或 Mermaid/whiteboard |
| 例子 | 紧跟对应结论后面，服务于说明，不单独堆素材 |

## Writing Rules

- 保留原文事实、指标、限制和归因；不确定的信息标成“待确认”，不要补脑。
- 用户只提供素材和大致位置时，主动补齐文档骨架、过渡句和表达形式；不要把素材按原顺序机械搬运。
- 把“背景很长才有结论”改成“结论一句话 + 关键依据 + 展开说明”。
- 把空泛动词换成可执行动作：例如“优化体验”改为“减少二次确认步骤”。
- 面向评估结果时，正文保留严谨但不要像审稿批注；不要主动加入“待统一口径”等旁白，除非用户要求审阅问题。
- 结论区的关键数字可四舍五入到 3 位有效小数，并补充 delta / 百分比；数据表保留原始精度。
- “下一步”写成 2-3 条可执行方向，优先使用编号列表；不展开成冗长方案。
- 长句拆短；一个段落只服务一个观点。
- 中英文标点跟随原文主语言；代码/API/schema 保持英文。

## Common Mistakes

- 只润色措辞，没有把结论移到最前面。
- 只整理素材，没有把它重组成一份完整文档。
- 只列出评论，没有把评论吸收到正文、表格、结论或下一步里。
- 机械吸收评论：只新增评论内容，没有检查它与原文是否重复、冲突，或是否要求重排章节和删除旧 raw 素材。
- 新增指标/维度后只改局部段落，没有联动更新综合公式、结果口径、顶部结论和下一步，导致正文自相矛盾。
- 删除、清空或整块重写带评论的段落、标题、表格单元、代码块，导致评论丢失或变成已删除内容；正确做法是保留评论对应内容，再在周边融合改写。
- 表格只改内容，没有设置 `<thead>/<th>` 和指定浅蓝背景 `rgb(225,234,255)`。
- 把普通术语、模型名、评估方法名、数字都做成 code，导致正文像日志而不是文档。
- 在结论块里堆太多 bullet，或把每个数字都 code；结论应该先像汇报，再像附录。
- 把 `snake_case`、参数、命令当普通文本，导致扫描困难。
- 为了好看过度使用画板、提示块和高亮；版式服务阅读，不服务装饰。
- 没读 `lark-doc` 的 XML/update 规则就直接拼命令，容易写坏块结构。
