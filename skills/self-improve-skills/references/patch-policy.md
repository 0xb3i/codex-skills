# 自改进补丁策略

## 默认直接修改

凡是直接影响目标 Skill 成功率、正确性或效率，且本轮证据足够支持的必要改进，默认直接写最小补丁并验证；不要把明确问题留给用户决定。

## 可以自动修改

- 修正明显过时、错误或互相矛盾的 Skill 文案。
- 增加本轮已验证的边缘 case、失败恢复路径或更简单路径。
- 改善触发描述，让 Skill 更容易在正确场景被使用。
- 给已有脚本补一个小的错误处理、参数说明或最小自检。
- 删除明显重复、误导或与实际工具不符的说明。

## 只记录候选，不自动修改

- 只是潜在影响、长期优化、隐蔽风险或不确定泛化，还没有直接影响当前 Skill 执行。
- 改变安全边界、审批策略、权限模型或生产操作流程。
- 写入密钥、token、账号、完整内部配置或无法脱敏日志。
- 大幅重组 Skill、引入新依赖、增加后台服务、hook、daemon 或数据库。
- 修改多个无关 Skill，或把一次偶发失败提升成通用规则。
- 用户的真实意图不清楚，且误改会影响后续任务。

记录候选后，在最终回复里告诉用户：候选是什么、为什么没有当下自动改、是否建议现在迭代。

## 验证最低要求

- 修改任何 Skill 后，运行 `quick_validate.py <skill-folder>`。
- 修改脚本后，运行该脚本的最小 smoke test；没有测试入口时加一个轻量 `--help` 或 `__main__` 自检。
- 修改复杂流程后，优先用一个独立子 agent 做 forward-test，但不要泄露预期答案或你的诊断。

## 写法偏好

- 一句能解决就一句，不新增抽象。
- 根因修在共享入口，不在每个调用处补丁。
- 记录“为什么现在要改”，不要记录完整对话流水。
- 明确保留已知上限，例如：`ponytail: 只处理单个 Skill；跨 Skill 归纳等出现 10 条以上候选再做。`
